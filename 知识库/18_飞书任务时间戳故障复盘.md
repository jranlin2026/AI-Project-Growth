# 飞书任务时间戳故障复盘

## 一句话结论

飞书任务 API 的 `due.timestamp` 必须使用 13 位毫秒级时间戳。如果误传 10 位秒级时间戳，飞书会把截止时间显示成 1970 年。

## 事故现象

2026-07-05 创建 Day3 飞书任务后，飞书任务列表显示：

```text
截止时间：1970年1月21日 23:20
```

受影响任务：

1. `2026/07/05周日 AI商业IP增长项目 Day3 作战任务：补齐 S002 闭环`
2. `N哥：12:00 前确认或补拍 S002《别再收藏 AI 工具了》`
3. `小彭：17:30 前剪辑并发布 S002`
4. `小彭：18:00 前发布朋友圈文案并反馈私域数据`
5. `2026/07/04 市场素材提交测试：只交链接、文案、评论截图`

## 根因

两个脚本把飞书任务截止时间写成了秒级时间戳：

```js
String(Math.floor(date.getTime() / 1000))
```

或：

```js
String(Math.floor(Date.UTC(...) / 1000))
```

飞书任务接口按毫秒级时间戳解析 `due.timestamp`，因此 10 位秒级时间戳会被当成 1970 年附近的毫秒值。

## 正确写法

必须传 13 位毫秒级时间戳：

```js
String(date.getTime())
```

或：

```js
String(Date.UTC(year, month - 1, day, hour - 8, minute, 0, 0))
```

任务字段示例：

```js
due: {
  timestamp: String(new Date("2026-07-05T17:30:00+08:00").getTime()),
  is_all_day: false
}
```

## 快速识别方法

读取任务清单：

```powershell
node scripts/feishu/list-tasklist-tasks.mjs
```

检查返回值：

```json
"due": {
  "timestamp": "1783243800000"
}
```

判断规则：

| 时间戳位数 | 含义 | 是否正确 |
| --- | --- | --- |
| 10 位 | 秒级 | 错误 |
| 13 位 | 毫秒级 | 正确 |

如果飞书界面显示 1970 年，优先检查 `due.timestamp` 是否为 10 位。

## 后续脚本规范

所有创建或更新飞书任务的脚本必须遵守：

1. `due.timestamp` 统一使用毫秒级。
2. 函数名建议写成 `shanghaiTimestampMs`，避免误解。
3. 创建任务后必须读取任务清单验证返回值。
4. 验证时必须检查 `due.timestamp` 是否为 13 位。
5. 如果显示 1970 年，不要重新创建任务，优先 PATCH 修正 `due` 字段。

## 推荐工具函数

```js
function shanghaiTimestampMs(date, hour, minute = 0) {
  const parts = new Intl.DateTimeFormat("en-CA", {
    timeZone: "Asia/Shanghai",
    year: "numeric",
    month: "2-digit",
    day: "2-digit"
  }).formatToParts(date);
  const map = Object.fromEntries(parts.map((part) => [part.type, part.value]));
  return String(Date.UTC(Number(map.year), Number(map.month) - 1, Number(map.day), hour - 8, minute, 0, 0));
}
```

## 经验结论

飞书任务创建后不能只看 API 成功返回。必须做二次验证：

```text
创建任务
  -> 读取任务清单
  -> 检查父任务存在
  -> 检查子任务存在
  -> 检查描述不是旧模板
  -> 检查 due.timestamp 是 13 位
  -> 再通知任务创建完成
```
