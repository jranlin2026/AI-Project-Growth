# 飞书任务、群消息与多维表格分工

> 更新口径：本文保留飞书工具分工的基础原则。实际每日派工以 [[11_三人协作每日SOP]]、[[12_每日执行包机制]]、[[15_AI与人协同增长SOP]] 为准；新版不再把阿浩/小彭固定成“编导 A/B”，而是按当天主责和复核分配。朋友圈和直播课推广也属于正式内容任务，不能只发在群里不记录。

## 一句话结论

飞书任务负责“谁今天要完成什么”，群消息负责“提醒大家看任务和交日报”，多维表格负责“沉淀日报、视频数据和复盘依据”。

## 推荐组合

不要只用群消息，也不要只用飞书任务。

最佳组合：

```text
飞书任务 = 派工与状态追踪
飞书群消息 = 通知与提醒
飞书多维表格 = 日报、视频数据、朋友圈数据、线索数据和复盘数据库
Codex = 生成任务、分析数据、安排明天
```

## 为什么不能只用群消息

群消息适合广播，但不适合管理任务。

问题：

- 消息容易被刷走
- 谁负责什么不够清晰
- 完成状态不好追踪
- 历史任务不好汇总
- 组员容易只回复“收到”，但没有交付

## 为什么要用飞书任务

飞书任务适合作为正式派工系统。

适合管理：

- 负责人
- 截止时间
- 子任务
- 完成状态
- 任务评论
- 今日待办
- 我负责的任务

## 为什么还需要多维表格

飞书任务适合管理执行，但不适合作为增长数据仓库。

短视频增长项目需要结构化记录：

- 每条视频播放量
- 点赞数
- 评论数
- “实战地图”评论数
- 私信数
- 进群人数
- 老板型线索
- 真实业务问题
- 最好视频和最差视频
- 每日复盘结论

这些数据更适合放在多维表格，方便 Codex 读取、筛选、统计和复盘。

## 第一版工作流

1. Codex 08:00 生成今日作战计划。
2. 系统在飞书任务里创建当天任务和子任务。
3. 飞书群发一条提醒：今日任务已创建，请查看任务清单。
4. 组员白天执行任务。
5. 17:00 飞书群提醒提交日报。
6. 组员在多维表格填写完成情况和视频数据。
7. Codex 读取多维表格，分析完成情况。
8. Codex 生成第二天任务。

## 任务拆分方式

每天创建一个父任务：

```text
2026-07-03 AI商业IP增长项目 Day X 作战任务
```

父任务下创建子任务：

| 子任务 | 负责人 | 截止时间 |
| --- | --- | --- |
| N哥：按完整脚本文档完成今日出镜拍摄 | N哥 | 17:00 |
| 当日主责：按执行包完成拍摄协助、剪辑/发布、最低反馈 | 阿浩或小彭 | 17:30 |
| 当日复核：检查成片口径、标题/封面/CTA，补明日选题 | 阿浩或小彭 | 17:30 |
| 数据复盘：填写视频数据表初填 | 当日数据负责人 / 数据分析智能体 | 17:30 |

排班规则：

- 工作时间：09:00-18:00。
- 小彭每周六休息，周六不分配人工执行任务。
- 阿浩每周日休息，周日不分配人工执行任务。
- 隔夜数据次日 09:30 补充。

## 群消息模板

### 08:00

```text
今日作战任务已创建。

请各负责人进入飞书任务查看自己负责的事项：
- N哥：按完整脚本文档出镜
- 当日主责：拍摄协助、剪辑/发布、最低反馈
- 当日复核：检查成片口径，补明日选题

17:30 前至少在任务评论或群里反馈完成情况；发布后补视频数据表。
```

### 17:00

```text
请提交今日日报。

要求：
1. 更新飞书任务状态
2. 在多维表格填写完成情况
3. 填写每条视频数据
4. 标记老板型评论和真实业务问题

未填写视为今日任务未闭环。
```

## 是否可以不要多维表格

如果只是管理“今天谁做什么”，可以不用多维表格。

但本项目是增长项目，需要长期复盘内容、账号、钩子、线索和成交机会，因此多维表格必须保留。

结论：

> 飞书任务管执行，多维表格管数据，群消息管提醒。
